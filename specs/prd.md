# PRD — Adaptive Tool-Building CLI Agent

작성일: 2026-06-01
상태: 설계 확정, 구현 착수 전

## 1. 개요

자연어로 들어온 작업을 받아, 그 작업을 풀 도구를 에이전트가 직접 만들어 실행하고 결과를 확인하는 CLI 에이전트를 만든다. 기성 에이전트 프레임워크에 기대지 않고 제어 루프를 손으로 구현해, LLM 호출부터 도구 실행과 재계획까지의 흐름이 코드에 그대로 드러나게 한다.

핵심은 두 가지 적응이다. 능력의 적응은 없는 도구를 그 자리에서 만들어 다음 요청부터 다시 쓰는 것이고, 컨텍스트의 적응은 대화가 길어져도 모델 입력의 앞부분을 안정적으로 유지하면서 변화분만 더하는 것이다.

## 2. 문제

자연어 지시를 받아 코드를 만들고, 실행하고, 결과를 확인한 뒤 다시 고치는 루프를 직접 다룰 수 있어야 한다. 이때 풀어야 할 문제는 다음과 같다.

- 미리 정해둔 도구만으로는 임의의 작업을 감당할 수 없다. 에이전트가 필요한 도구를 스스로 판단해 만들어야 한다.
- 모델이 짠 코드는 신뢰할 수 없다. 실행을 격리하지 않으면 위험하다.
- 한 번에 맞는 코드를 짜지 못한다. 실패를 피드백으로 되먹여 고치는 폐루프가 필요하다.
- 요청이 모호하거나 되돌리기 어려운 작업이면 임의로 진행하면 안 된다. 사람에게 물어야 한다.
- 한 번 만든 도구를 매번 새로 짜면 낭비다. 승인된 도구는 다음 세션에서도 재사용해야 한다.
- 대화가 길어지면 맥락이 과하게 쌓여 품질이 떨어진다. 컨텍스트를 관리해야 한다.

## 3. 해결책

위 문제를 다음 구현으로 푼다.

- 외부 추상화 없이 작성한 제어 루프가 LLM 호출, action 파싱, 도구 실행, 결과 관찰, 재계획, 종료를 직접 처리한다.
- 매 턴 LLM은 행동 하나를 JSON으로 반환한다. JSON repair와 schema 검증으로 깨진 출력을 복구하거나 다시 요청해 약한 모델에서도 견딘다.
- 도구 생성과 자가수정을 폐루프로 둔다. 실패하면 오류를 관찰해 코드를 고친 뒤 다시 실행한다.
- 생성 코드는 subprocess로 격리해 timeout과 경로·네트워크 제한 아래에서 돌린다.
- ALLOW, DENY, ASK_USER 세 단계 권한 게이트로 위험하거나 모호한 행동을 거른다.
- 승인된 도구는 메타데이터와 함께 skill 라이브러리에 저장하고, 다음 세션에서 이름과 설명만으로 다시 등록한다.
- 컨텍스트는 누적, 요약 compaction, 파일 영속의 세 층으로 관리한다.
- 검증은 실행 성공이 아니라 결과 상태 확인으로 둔다.

## 4. 주요 요구사항

### 기능 요구사항

- F1. 에이전트가 문제를 분석해 필요한 도구를 직접 생성하고 실행한다.
- F2. 내장 도구를 갖춘다. 파일 읽기·쓰기·목록, 제한된 Python 실행, 도구 생성·수정, 문서·스키마 조회, 사용자 질문.
- F3. 실행 실패 시 오류를 분석해 도구 코드를 고치고 다시 실행한다.
- F4. 대화 컨텍스트를 유지하고, 길어지면 핵심을 보존하며 요약으로 접는다.
- F5. 모호한 요청에는 실행 전에 사용자에게 추가 정보를 묻는다.
- F6. 사용자가 승인한 도구를 영속 저장하고 다음 세션에서 재사용한다.
- F7. 결과가 의도한 상태가 되었는지 다시 읽어 검증한다.
- F8. 모든 실행을 로그로 남기고, 선택적으로 외부 모니터링으로 내보낸다.

### 비기능 요구사항

- N1. 생성 코드 실행을 격리한다. timeout, 작업 디렉터리 제한, 네트워크 기본 차단, 출력 크기 제한.
- N2. 무한 루프와 비용 폭주를 막는다. 최대 반복과 재시도 상한.
- N3. provider 비종속 LLM 접근. 키 없이 돌릴 수 있는 로컬 경로를 1순위로 안내.
- N4. 약한 모델과 로컬 모델에서도 견디는 출력 파싱.
- N5. 입력의 앞부분을 안정적으로 유지해 캐시 적중을 높인다.
- N6. 공개 산출물 어디에도 작업 배경이나 출처를 드러내지 않는다.

## 5. 제약사항

- 에이전트 추상화 라이브러리를 쓰지 않는다. 제어 루프를 직접 구현한다.
- CLI 기반으로 만든다. 별도 웹이나 GUI는 두지 않는다.
- 생성 코드는 신뢰할 수 없는 코드로 간주한다.
- 특정 모델 선택 자체는 목적이 아니다. 선택한 모델에 맞는 견고한 설계가 중요하다.
- 외부 모니터링 연동은 선택이며, 끄더라도 에이전트가 정상 동작해야 한다. 모니터링 의존성이 핵심 경로를 막지 않는다.

## 6. 핵심 파이프라인

한 번의 사용자 요청이 처리되는 흐름이다.

1. 입력 수집. 사용자 요청과 현재 컨텍스트, 사용 가능한 도구의 이름과 설명을 모은다. 도구의 전체 코드와 스키마는 이 단계에 넣지 않는다.
2. 계획. LLM이 다음 행동 하나를 JSON으로 반환한다. respond, ask_user, call_tool, create_tool, update_tool, finish 중 하나다.
3. 검증과 복구. 반환값을 JSON repair와 schema 검증으로 확인한다. 깨지면 오류를 붙여 다시 요청한다.
4. 분기 실행.
   - ask_user면 멈추고 사용자에게 묻는다.
   - call_tool이면 권한 게이트를 통과한 뒤 도구를 실행한다. 생성 도구는 격리 실행한다.
   - create_tool과 update_tool이면 도구 코드를 만들거나 고친다.
   - respond와 finish면 답하거나 종료한다.
5. 관찰. 실행 결과와 오류를 observation으로 누적한다.
6. 결과 상태 검증. 의도한 상태가 되었는지 다시 읽어 확인한다. 어긋나면 실패 사유를 observation으로 넘긴다.
7. 반복 또는 종료. 종료 조건이나 최대 반복에 도달할 때까지 2번으로 돌아간다.
8. 영속화 제안. 작업을 마치면 생성 도구를 저장할지 묻고, 승인 시 skill로 저장한다.

각 단계의 입출력과 결정은 로그로 남긴다.

## 7. 로그와 모니터링

로그는 두 층으로 둔다. 로컬 로그는 항상 켜지고, 외부 모니터링은 교체 가능한 익스포터로 선택 연동한다. 목적은 두 가지다. 하나는 실행을 사람이 읽고 디버깅하는 것이고, 다른 하나는 누적된 실행 기록으로 프롬프트와 루프를 튜닝하는 것이다.

### 7.1 로컬 로그

- CLI 화면에는 Rich로 사람이 읽기 좋은 진행 로그를 보여준다. 단계, 선택한 action, 도구 실행, 검증 결과.
- 동시에 구조화 로그를 JSONL 파일로 남긴다. 한 줄이 하나의 이벤트다.
- 한 번의 요청 처리를 하나의 trace로 묶고, 그 안의 LLM 호출과 도구 실행을 span으로 둔다. trace id와 span id로 연결한다.

남기는 필드(예):

- trace id, span id, parent span id
- 이벤트 종류: llm_call, tool_call, tool_create, tool_update, verify, ask_user, policy_decision
- 타임스탬프와 소요 시간
- LLM 호출이면 model, 입력·출력 토큰 수, 캐시 적중 여부, action 종류, 파싱 성공 여부와 재시도 횟수
- 도구 실행이면 도구 이름, 종료 코드, timeout 여부, 출력 크기, 잘림 여부
- 권한 결정이면 ALLOW, DENY, ASK_USER 중 무엇이었는지와 사유
- 검증이면 통과 여부와 실패 사유
- 자가수정이면 몇 번째 수정 시도인지
- 오류면 종류와 메시지 요약

민감 정보와 비밀 값은 로그에 남기지 않는다. 큰 출력은 잘라 저장한다.

### 7.2 외부 모니터링 익스포터

- 모니터링 연동은 하나의 인터페이스 뒤에 둔다. 기본 구현은 아무 데도 보내지 않는 no-op이고, 환경 변수로 켜면 외부 모니터링으로 내보낸다.
- 1차 대상은 Langfuse처럼 trace, span, generation, score 개념을 가진 LLM 관측 도구다. 로컬 trace와 span 모델을 그 개념에 그대로 대응시킨다.
- 내보내는 항목은 로컬 로그와 같은 단위다. 한 요청이 한 trace, 그 안의 LLM 호출과 도구 실행이 span, LLM 호출은 generation으로 둔다.
- 평가 점수를 score로 붙인다. 검증 통과 여부, 자가수정 횟수, 한 작업당 LLM 호출 수, 캐시 적중률 같은 지표를 score로 기록해 튜닝 근거로 쓴다.
- 익스포터는 비동기 또는 비차단으로 동작하며, 실패해도 에이전트 핵심 경로를 막지 않는다.

### 7.3 튜닝 활용

누적된 trace로 다음을 본다.

- 작업당 평균 LLM 호출 수와 토큰 사용량
- 캐시 적중률 추이
- 자가수정 루프가 도는 빈도와 평균 반복 수
- action 파싱 실패율과 재요청 빈도
- 검증 실패가 잦은 작업 유형

이 지표로 프롬프트, 도구 설명, 종료 조건, compaction 임계를 조정한다.

## 8. 주요 개발 feature list

### MVP

1. provider 비종속 LLM 클라이언트. 로컬 OpenAI 호환과 호스팅을 환경 변수로 교체.
2. 직접 구현한 제어 루프와 종료 조건, 반복 상한.
3. JSON action 프로토콜. repair, schema 검증, 재요청.
4. 내장 도구: 파일 읽기·쓰기·목록, 제한된 Python 실행, 도구 생성·수정, 문서·스키마 조회, 사용자 질문.
5. 격리 실행 sandbox: subprocess, timeout, 경로 제한, 네트워크 차단, 출력 제한.
6. 자가수정 폐루프.
7. 권한 게이트와 모호성 질문.
8. skill 영속 저장과 다음 세션 재로딩. 이름·설명 상시 노출, 본문 지연 로딩.
9. 세 층 컨텍스트 관리와 compaction.
10. 결과 상태 검증.
11. 로컬 구조화 로그와 Rich 진행 출력.
12. 모니터링 익스포터 인터페이스와 no-op 기본 구현.
13. 상태형 객체 트리 예제 시나리오와 샘플 데이터.
14. README: 실행 방법, 환경과 LLM 설정, 설계 결정, 한계와 개선 방향.

### Stretch

15. Langfuse 익스포터 구현과 score 기록.
16. 복잡도 판단과 작업 목록 위임.
17. 작업에 맞춘 동적 워크플로 구성.
18. compaction 자동화 고도화.
