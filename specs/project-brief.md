# 프로젝트 브리프

## 목적

자연어로 들어온 임의의 작업을 받아, 그 작업을 풀기 위한 도구를 에이전트가 직접 만들어 실행하고 결과를 확인하는 CLI 에이전트를 만든다. 기성 에이전트 프레임워크에 기대지 않고 제어 루프를 손으로 구현해서, LLM 호출부터 도구 실행과 재계획까지의 흐름이 코드에 그대로 드러나게 한다.

핵심 가치는 두 가지 적응이다. 하나는 능력의 적응으로, 없는 도구를 그 자리에서 만들어 다음 요청부터 다시 쓴다. 다른 하나는 컨텍스트의 적응으로, 대화가 길어져도 모델 입력의 앞부분을 안정적으로 유지하면서 변화분만 더한다.

## 핵심 개발 내용 (필수 구현)

- 직접 구현한 제어 루프: LLM 호출, action 파싱, 도구 실행, 결과 관찰, 재계획, 종료 조건을 외부 추상화 없이 작성한다.
- JSON action 프로토콜: 매 턴 LLM이 행동 하나를 JSON으로 반환한다. JSON repair와 schema 검증으로 깨진 출력을 복구하거나 다시 요청해서 약한 모델에서도 견디게 한다.
- 내장 도구 일습: 파일 읽기·쓰기·목록, 제한된 Python 실행, 도구 생성·수정, 문서·스키마 조회, 사용자 질문.
- 도구 생성과 자가수정: 필요한 도구를 코드로 만들어 실행하고, 실패하면 오류를 관찰해 코드를 고친 뒤 다시 실행하는 폐루프.
- 격리 실행: 생성 코드를 같은 프로세스에서 직접 돌리지 않고 subprocess로 분리한다. timeout, 작업 디렉터리 제한, 네트워크 기본 차단, 출력 크기 제한을 둔다.
- 권한 게이트와 사람 개입: ALLOW, DENY, ASK_USER 세 단계로 위험하거나 모호한 행동을 거른다. 요청이 모호하면 실행하지 않고 되묻는다.
- 도구 영속화: 사용자가 승인한 도구를 메타데이터와 함께 skill 라이브러리에 저장하고, 다음 세션에서 이름과 설명만으로 다시 등록해 재사용한다.
- 컨텍스트 관리: 대화 누적을 기본으로 두고, 길어지면 핵심 사실을 보존하며 요약으로 접는다. 작업 요청과 결과 요약은 파일로도 남긴다.
- 결과 상태 검증: 실행이 끝났는지가 아니라 의도한 상태가 되었는지를 다시 읽어 확인한다.

## 제약 사항

- 에이전트 추상화 라이브러리를 쓰지 않는다. 제어 루프를 직접 구현한다.
- CLI 기반으로 만든다. 별도 웹이나 GUI는 두지 않는다.
- 생성 코드는 신뢰할 수 없는 코드로 간주하고 격리 실행을 전제로 한다.
- LLM은 provider 비종속으로 접근하며, 키 없이 돌릴 수 있는 로컬 경로를 1순위로 안내한다. 특정 모델 선택 자체는 목적이 아니다.
- 공개 저장소에는 이 작업의 배경이나 출처를 드러내는 표현을 어디에도 남기지 않는다. 문제와 예제는 일반 용어로만 기술한다.

## 최종 결과물

- 위 필수 구현을 갖춘 CLI 에이전트 소스 코드.
- 상태를 가진 객체 트리를 다루는 예제 시나리오와 샘플 데이터.
- 단위 테스트와 통합 테스트. LLM 호출은 결정적 테스트를 위해 대체 구현으로 둔다.

## 예상되는 성공 시나리오

### 시나리오 1 — 데이터 질의

사용자가 JSON 데이터에서 특정 조건을 만족하는 항목과 평균값을 묻는다. 에이전트는 파싱, 필터, 집계가 필요하다고 판단해 도구를 만들어 실행하고, 결과를 다시 확인한 뒤 답한다.

### 시나리오 2 — 실패와 자가수정

생성한 도구에 버그가 있어 실행이 실패한다. 에이전트가 오류 메시지를 관찰하고 원인을 분석해 코드를 고친 뒤 다시 실행해 성공한다.

### 시나리오 3 — 모호한 요청

사용자가 "데이터 정리해줘"처럼 모호하게 요청한다. 에이전트는 바로 실행하지 않고 어떤 데이터를 어떤 방식으로 정리할지 되묻는다. 사용자가 구체적인 입력을 주면 그때 도구를 만들어 실행한다.

### 시나리오 4 — 도구 저장과 재사용

작업을 마친 뒤 에이전트가 방금 만든 도구를 저장할지 묻는다. 사용자가 승인하면 skill로 저장한다. 다음 세션에서 비슷한 작업이 들어오면 같은 도구를 새로 만들지 않고 다시 불러 쓴다.

### 시나리오 5 — 상태를 가진 객체 트리

작업 영역에 계층적 객체 트리가 JSON으로 놓여 있다. 사용자가 그 트리를 조건에 맞게 바꾸거나 질의하도록 요청한다. 에이전트는 트리 스키마와 허용 연산을 먼저 조회한 뒤 도구를 만들어 트리를 바꾸고, 다시 읽어 의도한 상태가 되었는지 검증한다. 성공하면 그 도구를 저장할지 묻는다.

### 시나리오 6 — 정책 거부

사용자가 작업 영역 밖 경로에 파일을 쓰라고 요청한다. 에이전트가 행동을 제안하더라도 런타임 권한 게이트가 경로 이탈을 DENY로 막고, 파일이 생성되지 않았음을 검증한다.

### 시나리오 7 — 후속 턴 컨텍스트

사용자가 이전 턴에서 필터링한 결과를 후속 요청에서 다시 사용하라고 지시한다. 에이전트는 컨텍스트에서 이전 결과를 회수해 새 산출물을 만들고, 파일 쓰기 승인과 결과 상태 검증을 거친다.
